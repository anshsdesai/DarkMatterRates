### 6. QCDark2 SHM performance

Status: fixed in the current follow-up pass.

Original problem: the QCDark2 engine was slow for `halo_model='shm'` because it
did not use the existing fast halo-file/interpolation path. In
`DMeRates/engines/dielectric.py`, `_resolve_eta` dispatched:

- `imb` -> `_eta_imb_bare(...)`, which used `eta_MB_tensor` and was fast.
- `shm`, `tsa`, `dpl` -> `_eta_scalar_bare(...)`, which wrapped scalar halo
  functions with `np.vectorize`.

`np.vectorize` is only a Python loop convenience wrapper. It evaluates
`etaSHM`/`etaTsa`/`etaDPL` one grid point at a time over the full `(N_q, N_E)`
array, which is why `test_qcdark2_shm_halo_runs` takes so long.

Before QCDark2 hookup, the legacy path was fast because `DMeRate.setup_halo_data`
loaded or generated halo files and `get_halo_data` used `FileHaloProvider` /
interpolation over tensors. The QCDark2 engine bypasses that machinery and only
receives a `DM_Halo_Distributions` object.

Implemented fix:

- Added an optional vectorized `eta_provider` argument to QCDark2
  `compute_dRdE`.
- Direct `compute_dRdE(..., halo_model='shm'/'tsa'/'dpl')` now builds the same
  file-backed/interpolated halo provider used by the legacy path.
- `DMeRate.calculate_qcdark2_rates(...)` and
  `DMeRate.calculate_qcdark2_spectrum(...)` now call `setup_halo_data(...)` for
  non-`imb`/non-`srdm` halos and pass a bridge to `get_halo_data(...)`.
- Added `test_dmerate_qcdark2_shm_uses_vectorized_halo_path`.
