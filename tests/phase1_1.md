# Phase 1.1 — Packaging via pyproject.toml

## What Was Done

Added a minimal `pyproject.toml` at the repo root so the project can be installed in editable mode with `pip install -e .`, preserving the existing `import DMeRates` public name without moving or renaming any directories.

---

## File Created

**`pyproject.toml`** (repo root)

```toml
[build-system]
requires = ["setuptools>=68", "wheel"]
build-backend = "setuptools.build_meta"

[project]
name = "dmerates"
version = "0.1.0"
...
```

The distribution name (`dmerates`) is lowercase per PyPI convention; the installed Python package name remains `DMeRates` because `[tool.setuptools] packages = ["DMeRates"]` tells setuptools to discover the directory exactly as-is.

## Package Data

```toml
[tool.setuptools.package-data]
DMeRates = ["p100K.dat", "Rates/*.dat"]
```

`p100K.dat` is the 100K silicon ionization probability table required at runtime. `Rates/*.dat` captures pre-calculated rate files generated by `generate_dat()`.

## Non-Obvious Findings

### 1. `torchinterp1d` is a local git submodule, not a PyPI package

`torchinterp1d` is declared as a git submodule (`path = torchinterp1d`, pointing to a fork at `ande8412/torchinterp1d`) and was already installed into the venv in editable mode:

```
Location: .venv/lib/python3.13/site-packages
Editable project location: .../DarkMatterRates/torchinterp1d
```

Adding it to `pyproject.toml` dependencies as `torchinterp1d @ file:torchinterp1d` would work with modern pip but breaks silently for anyone who clones without `--recurse-submodules`. The line is left as a comment with instructions. Users must run `pip install -e torchinterp1d/` separately if installing from a fresh clone.

### 2. `torch` CUDA variant cannot be expressed in pyproject.toml

The venv has `torch==2.6.0+cu126`, a CUDA-specific build available only from PyTorch's own wheel index (`https://download.pytorch.org/whl/cu126`), not from PyPI. Specifying a local-version suffix in `dependencies` would cause `pip install -e .` to fail on machines that use a different CUDA version or CPU-only torch. The dependency is listed as `torch>=2.0` with a comment directing users to the appropriate install command.

### 3. `backends/` subdirectory contains only `__pycache__`

`DMeRates/backends/` exists but holds no Python source files. It is not included as a package. If it becomes a real subpackage in a future phase, add `"DMeRates.backends"` to `[tool.setuptools] packages`.

### 4. AST scan surfaced `Constants` as apparent third-party import

An AST walk over `DMeRates/*.py` reported `Constants` alongside true third-party packages. All actual `Constants` imports in the package are relative (`from .Constants import *`), so this was a scanner artifact. No absolute `import Constants` exists in the runtime code; the false positive was ignored.

## What Is Deferred

- Moving `torchinterp1d` from a git submodule to a proper PyPI or VCS dependency (`torchinterp1d @ git+https://...`) is a packaging improvement for a later phase, once the fork relationship is clarified.
- Version pinning (`torch==2.6.0`, `numpy==2.2.4`, etc.) is left to the environment's `requirements.txt`. The `pyproject.toml` states only minimum bounds to remain installable across environments.
- The `[dft]` optional group is a stub; QCDark2 DFT-generation dependencies are not yet determined.
